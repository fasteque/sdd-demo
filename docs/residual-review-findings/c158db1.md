# Known Residuals — c158db1 (GET /assets pagination)

Recorded from the `ce-code-review` pass on `feat(assets): add GET /assets paginated list endpoint`.
Both findings were reviewed and accepted by the user rather than fixed immediately, since each depends on a design decision already made (or requiring a contract change) rather than a mechanical fix.

## 1. No upper bound on `size` query parameter

- **File:** `src/main/kotlin/ch/fasteque/sdd_demo/AssetController.kt:65`
- **Category:** robustness
- **Finding:** `size` is validated for a lower bound (`>= 1`) but has no upper bound before being passed to `PageRequest.of`.
- **Failure scenario:** `GET /assets?size=2000000000` passes validation and forces the app to fetch and serialize an unbounded number of documents in one response, risking memory exhaustion or a very slow request.
- **Why deferred:** Already named as an accepted trade-off in `openspec/changes/list-assets-pagination/design.md`'s Risks section ("Acceptable for this learning sandbox's scale; revisit with a max-size cap if this becomes a real concern"). No new information changed that call.

## 2. `AssetPage` duplicates Spring's `PagedModel<T>`

- **File:** `src/main/kotlin/ch/fasteque/sdd_demo/AssetController.kt:21`
- **Category:** reuse
- **Finding:** The hand-rolled `AssetPage` DTO (content/page/size/totalElements/totalPages) duplicates `org.springframework.data.web.PagedModel<T>`, which ships in spring-data-commons (already a transitive dependency) and solves the same `PageImpl` JSON-serialization-instability problem `design.md` cites as the reason for adding a custom class — `design.md` just never considered `PagedModel` as an option.
- **Failure scenario:** Future paginated endpoints in this codebase either copy this bespoke shape again or inconsistently adopt `PagedModel`, producing two different pagination JSON contracts across the API.
- **Why deferred:** Switching to `PagedModel<Asset>` changes the JSON response shape (nested `content`/`page` object vs. today's flat fields), which the spec (`specs/asset-management/spec.md`) and tests already commit to. Worth considering before or alongside the next paginated endpoint, not a blocker for this one.
